## PRIORITY 2 — OpenClaw Ambiguities
*These prevent deterministic agent execution. Each "Claude determines..." statement needs a threshold or algorithm.*

---

### A1 — Design approval: no confidence threshold
**Spec location:** Part 13, Agent 6; Part 8, routing table
**Risk:** "Claude determines intent, not keyword matching" — Emily could say "I love it" conversationally and accidentally lock in a design. Build summaries go to Christopher immediately.
**Fix:**
> Design approval requires confidence ≥ 0.85. If Claude scores 0.50–0.84, respond: *"Just want to make sure — are you happy for me to lock this in and write up the build plan?"* and wait for explicit confirmation. Below 0.50 = not approval, continue the design session.

---

### A2 — Bed alias resolution: no algorithm
**Spec location:** Part 13, Agent 2 (Extraction); Part 5, bed schema (`aliases` array)
**Risk:** Agent 2 must map "the tomato place" or "west bed" to a canonical `bed_id`. If multiple beds have overlapping aliases, the wrong bed gets updated.
**Fix:**
> Resolution order: (1) Exact match against `aliases` array. (2) Case-insensitive match. (3) If still ambiguous, ask: *"Just want to make sure — are you talking about [bed-01] or [bed-03]?"* Never guess. Log unresolved aliases to `data/system/write_log.json` with `resolution: 'ambiguous'`.

---

### A3 — Medium-confidence writes: no surface channel or timeout
**Spec location:** Part 4, confidence thresholds
**Risk:** "Surface to user before saving" does not specify which Discord channel, which user is @mentioned, or what happens if no response is received.
**Fix:**
> Surface medium-confidence (0.50–0.79) observations as an @mention to the message author in the channel where the conversation occurred. Example: *"@emily I noticed [observation] — want me to remember that?"* If no response within 24 hours, discard the observation and log `outcome: 'expired_no_response'` to `write_log.json`. Never auto-save medium-confidence observations.

---

### A4 — Heartbeat "friend test": subjective, no rubric
**Spec location:** Part 13, Agent 3
**Risk:** "Would a knowledgeable friend actually say this, right now, in this way?" is not executable. OpenClaw cannot verify the output meets this standard.
**Fix:** Add worked examples to `prompts/agents/AGENTS.md` under Agent 3:

**PASS examples:**
- Treatment is 10 days overdue and the weather this week is ideal for it
- Emily mentioned a concern 2 weeks ago and there's been no update since
- A cold snap is forecast and a sensitive plant hasn't been covered

**FAIL examples:**
- Weather is normal and no tasks are overdue (nothing urgent to say)
- Same observation already posted in the last 7 days
- The information is purely factual and Emily didn't ask for it

---

### A5 — Heartbeat "max 2/week" has no tracking mechanism
**Spec location:** Part 13, Agent 3
**Risk:** Agent 3 runs every 30 minutes but has no counter. Without tracking, it will post more than 2 times per week.
**Fix:**
> Before every Heartbeat post, check `data/system/health_status.json` field `proactive_posts_this_week`. If ≥ 2, skip and log `skipped: true`. Reset this counter to 0 every Sunday at midnight. Add `proactive_posts_this_week: 0` and `proactive_posts_week_of: "ISO 8601 week start"` to the `health_status.json` schema in Part 5.

---

### A6 — `garden.md` regeneration trigger is undefined
**Spec location:** Part 4, extraction agent step 8 (line "garden.md regeneration")
**Risk:** If garden.md only regenerates on extraction but no conversation happens for 3 days, the hot context fed to every Claude call is stale.
**Fix:**
> Regenerate `garden.md` under two conditions:
> 1. After any extraction that writes a `BED_FACT` or `DECISION` event
> 2. Daily at 9:00 AM regardless of new extractions (to capture weather shifts, seasonal transitions)
>
> File path: `docs/garden.md`. Add `last_regenerated_at` front matter. If regeneration fails, use previous version and log to `health_status.json`.

---

### A7 — Race condition between Extraction agent (Agent 2) and Heartbeat (Agent 3)
**Spec location:** Parts 13, Agents 2 and 3
**Risk:** Extraction fires 60 seconds after conversation silence; Heartbeat runs every 30 minutes. If extraction writes a new `follow_up_required: true` event at 18:32 but Heartbeat ran at 18:30, the follow-up won't be surfaced until 19:00.
**Fix:**
> Heartbeat is idempotent — each 30-minute run re-evaluates all conditions from scratch against the current state of data files. Before posting, check: has Heartbeat already posted on this same `subject_id` in the last 24 hours? If yes, skip. Deduplication key: `channel + subject_id + event_category`. Store last-post timestamps in `data/system/health_status.json`.

---

### A8 — Timeline event deduplication: "approximate content" is not measurable
**Spec location:** Part 4, write rules
**Risk:** "Check for an existing entry with the same subject, type, and approximate content" — an AI agent cannot execute "approximate" without a defined threshold. Same observation could be logged repeatedly.
**Fix:**
> Deduplication window: **48 hours**. Within the window, an event is a duplicate if: same `subject_id` + same `event_type` + (exact text match OR text similarity > 0.85 by normalized fuzzy ratio). Outside the 48-hour window, always append as a new event — repeat observations over time are valid data.

---

### A9 — Photo confidence overwrite rule is non-numeric
**Spec location:** Part 4, "Photo observations can freely add to memory. It can only overwrite an existing entity fact if its confidence score is higher."
**Risk:** "Higher confidence" is ambiguous when user-confirmed facts don't have a numeric confidence value stored.
**Fix:**
> Assign implicit confidence values:
> - User-confirmed fact (author: emily or christopher, manually confirmed): **1.0**
> - Auto-saved high-confidence extraction: the stored numeric value (0.80–0.95)
> - Photo observation: capped at **0.95** regardless of Gemini output
>
> A photo observation (≤ 0.95) **never overwrites** a user-confirmed fact (1.0). Photo observations overwrite lower-confidence auto-saves only if `new_confidence > stored_confidence`.

### A10 — Voice session → extraction trigger is unspecified
**Spec location:** Part 15, post-session flow
**Risk:** "Orchestrator detects new pending voice session" — there is no mechanism defined. Without a trigger, voice session extractions never run.
**Fix:**
> Two options (choose one in Phase 13):
> - **Option A (recommended):** `voice_webhook.py` calls OpenClaw's trigger endpoint immediately after saving the session JSON, passing `session_id`. OpenClaw runs extraction as an async job.
> - **Option B (polling):** Dedicated cron runs every 2 minutes, scans `data/voice_sessions/YYYY/` for files with `extraction_status: 'pending'`, processes each.
>
> Add `extraction_status: 'pending' | 'complete' | 'failed'` to the voice session schema in Part 5.

| Command | Action |
|---|---|
| `!status` | Load `health_status.json`, format as readable summary, post to channel |

Route: Any message starting with `!` goes directly to Agent 8, bypassing mode classification.

### C3 — No Postgres backup strategy
**Risk:** The Docker volume at `slarti/db/data/` is not tracked by Git. A volume corruption means total data loss for all pgvector embeddings and structured data.
**Fix:** Add to `scripts/git_push.sh` (before the Git push):
```bash
# Daily Postgres backup
pg_dump -h localhost -U slarti slarti > /mnt/c/Openclaw/slarti/backups/db_$(date +%Y%m%d).sql
# Keep only last 14 backups
ls -t /mnt/c/Openclaw/slarti/backups/db_*.sql | tail -n +15 | xargs rm -f
```

Add `backups/` to `.gitignore` (SQL dumps can be large). Note: `backups/` is local-only; if the machine fails, the last GitHub push is the restore point.
